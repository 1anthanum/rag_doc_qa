## 一、这个项目能做什么？

用一句话概括：**你上传文档，用自然语言提问，AI 从你的文档中找到答案并告诉你来源。**

更具体地说，系统支持以下能力：

| 能力 | 说明 |
|------|------|
| 多格式文档 | PDF、Word (.docx)、纯文本 (.txt)、Markdown (.md) |
| 智能检索 | 向量语义搜索 + BM25 关键词搜索，融合取最优结果 |
| 4 种分块策略 | 固定长度、句子边界、递归切分、语义切分 |
| 查询优化 | 自动重写模糊问题、HyDE 假设文档生成、复杂问题分解 |
| 自纠错检索 | AI 自动评估检索结果，不满意就换个方式重新搜索 |
| 4 种问答模式 | 精确问答、摘要、对比、多轮对话 |
| 双 LLM 后端 | OpenAI（云端）或 Ollama（完全本地，无需 API Key） |
| 双界面 | Streamlit 网页 + FastAPI REST API |
| 配置驱动 | 所有功能都可通过一个 YAML 文件开关控制 |

## 二、核心概念：5 分钟理解 RAG

### 问题：AI 为什么需要"查资料"？

ChatGPT 这样的 AI 很聪明，但它只知道训练时见过的内容。你公司的内部文档、你昨天写的报告——它一无所知。

**RAG**（Retrieval-Augmented Generation，检索增强生成）的思路很直观：

```
                       传统 AI
问题 ──────────────────────────────→ AI 凭记忆回答（可能瞎编）

                       RAG
问题 ──→ 先从你的文档中搜索 ──→ 把搜到的内容 + 问题一起给 AI ──→ 有据可查的回答
```

打个比方：传统 AI 像一个博学但没读过你报告的专家；RAG 是先让这个专家翻阅你的报告，再回答。

### 三个阶段

整个系统像一条流水线，分三步：

```
📄 文档 ──→ 【摄入】切块 + 向量化 ──→ 💾 数据库
                                          ↓
❓ 提问 ──→ 【检索】找最相关的片段 ──────→ 🤖【生成】AI 基于片段回答 ──→ 💬 答案
```

**阶段一 · 摄入（Ingestion）：** 把文档切成小块，每块转换成一串数字（向量），存入数据库。

**阶段二 · 检索（Retrieval）：** 把你的问题也转成向量，在数据库中找到"数字上最接近"的文本块。

**阶段三 · 生成（Generation）：** 把找到的文本块和你的问题一起发给 AI，AI 基于这些内容生成回答。

> 如果你想深入了解每个阶段的技术细节，请阅读 [how-it-works.md](./how-it-works.md)。

## 三、安装与启动

### 前提条件

- Python 3.10 或更高版本
- 8 GB 以上内存（本地 embedding 模型约占 500 MB）
- （可选）OpenAI API Key，或安装 Ollama 用本地模型

### 安装步骤

```bash
# 1. 克隆项目
git clone https://github.com/<your-username>/rag-doc-qa.git
cd rag-doc-qa

# 2. 安装依赖
pip install -r requirements.txt

# 3. 设置 API Key（如果使用 OpenAI）
export OPENAI_API_KEY="sk-..."

# 4. 启动 Streamlit 界面
streamlit run app.py
```

浏览器会自动打开 `http://localhost:8501`。

### 完全离线模式（无需 API Key）

```bash
# 安装 Ollama：https://ollama.ai
ollama pull llama3.1

# 以本地模式启动
LLM_PROVIDER=ollama EMBEDDING_PROVIDER=local streamlit run app.py
```

### 启动 API 服务器

```bash
uvicorn src.api.endpoints:create_app --factory --reload --port 8000
```

API 文档自动生成在 `http://localhost:8000/docs`。

## 四、基础用法：上传文档 → 提问 → 获取回答

启动后，你会看到一个聊天界面。左侧是控制面板，右侧是对话区域。

### 第一步：上传文档

1. 在左侧 **📄 Document Upload** 区域点击上传
2. 支持 PDF、TXT、MD、DOCX 格式，可一次上传多个文件
3. 点击 **🔄 Index Documents** 按钮
4. 等待进度条完成——系统正在分块、向量化、存入数据库

左侧会显示 **Indexed Chunks** 数字，表示数据库中有多少个文本块可供搜索。

### 第二步：调整设置

在左侧 **⚙️ Settings** 区域：

- **Query Mode** — 选择问答模式（下一节详解）
- **Top-K Results** — 检索多少个文本块（默认 5，越大信息越全但可能引入噪声）
- **Temperature** — AI 回答的创造性（0 = 严格精确，1 = 自由发挥）

### 第三步：提问

在底部输入框输入你的问题，按回车。系统会：

1. 在数据库中搜索最相关的文本块
2. 把文本块 + 你的问题发给 AI
3. 显示 AI 的回答
4. 点击 **📎 Sources** 展开查看信息来源

## 五、四种问答模式详解

### QA（精确问答）— 默认模式

适合有明确答案的问题。系统会尽量给出简短、准确的回答并标注来源。

> 问："项目的交付日期是什么？"
> 答："根据项目计划（来源 3），交付日期为 2024 年 12 月 15 日。"

### Summarize（摘要）

适合需要概览的场景。系统会总结检索到的所有内容的要点。

> 问："帮我总结这份报告的核心发现"
> 答："报告主要发现包括三点：一是……二是……三是……（分别来自来源 1、3、5）"

### Compare（对比）

适合多文档或多角度对比。系统会对比不同来源的说法，指出异同。

> 问："这两份报告对市场趋势的预测有何不同？"
> 答："报告 A 认为市场将增长 15%（来源 2），而报告 B 持保守态度，预计仅增长 5%（来源 4）。主要分歧在于……"

### Conversational（多轮对话）

适合连续追问。系统会记住之前的问答，理解上下文。

> 问 1："项目预算是多少？"
> 答 1："预算为 500 万元。"
> 问 2："其中人力成本占比多少？"（系统知道"其中"指的是刚才提到的 500 万预算）
> 答 2："人力成本占总预算的 60%，即 300 万元。"

## 七、进阶：混合检索（Hybrid Search）

### 问题

纯向量搜索擅长理解语义（"预算"和"资金分配"意思相近），但可能漏掉精确关键词。比如你搜"ISO-27001"，向量搜索可能匹配到"安全标准"相关的段落，但不一定包含"ISO-27001"这个精确词。

### 解决方案

混合检索同时使用两种搜索引擎：

- **向量搜索（Dense）：** 理解语义，找意思相近的内容
- **BM25 搜索（Sparse）：** 关键词精确匹配，像传统搜索引擎

然后用 **Reciprocal Rank Fusion (RRF)** 算法把两者的结果合并排序。研究表明，这种组合比单独使用任一种方法**提高 15-25% 的检索准确率**。

```
           "ISO-27001 审计流程"
                │
     ┌──────────┼──────────┐
     ▼                     ▼
  向量搜索              BM25 搜索
  找到意思相近的       找到包含精确
  "安全审计"段落      "ISO-27001"的段落
     │                     │
     └──────────┬──────────┘
                ▼
           RRF 融合排序
           取最佳结果
```

### 如何开启

在 `configs/default.yaml` 中：

```yaml
retrieval:
  mode: hybrid           # 改为 hybrid

  hybrid_search:
    enabled: true
    dense_weight: 0.7    # 向量搜索权重（0-1）
    sparse_weight: 0.3   # BM25 权重（0-1）
    rrf_k: 60            # RRF 常数（默认 60，通常不需要改）
```

### 什么时候该用

- 文档包含大量专业术语、编号、型号、法规名称
- 用户经常搜索精确关键词
- 需要同时兼顾"精确匹配"和"语义理解"

## 八、进阶：语义分块（Semantic Chunking）

### 问题

默认的切分方式（递归切分）按字数或段落机械切分。这可能导致一个主题被切到两个不同的块里，或者两个不相关的话题被塞进同一个块。

### 解决方案

语义分块不看字数，而是看**内容是否在讲同一件事**。它的工作流程：

1. 把文本切成一个个句子
2. 对每个句子生成向量
3. 计算相邻句子的**余弦相似度**（两个向量的夹角越小，意思越接近）
4. 在相似度骤降的地方切分——那里就是话题转换点

```
句子 1: "项目预算为 500 万" ─── 相似度 0.85 ─── 句子 2: "其中人力成本占 60%"
                                                        │
                                                   相似度 0.2（骤降！在这里切分）
                                                        │
句子 3: "技术选型方面，我们采用微服务架构" ─── 相似度 0.78 ─── 句子 4: "每个服务独立部署"
```

### 如何开启

```yaml
ingestion:
  chunking:
    strategy: semantic           # 改为 semantic
    semantic_threshold: 0.5      # 相似度低于此值就切分（0-1，越低切得越少）
```

### 注意事项

- 语义分块需要对每个句子生成向量，**摄入速度会变慢**（大约慢 2-3 倍）
- 适合文档主题跨度大的场景（比如一份报告包含财务、技术、人事多个话题）
- 对于主题单一的短文档，默认的递归切分已经够用

## 九、进阶：查询优化（HyDE / 重写 / 分解）

### 问题

用户输入的问题往往很短、很模糊。比如"预算？"——系统很难从这两个字中理解你到底想知道什么。

### 三种优化策略

#### 策略一：查询重写（Rewrite）

让 AI 把模糊问题重写为具体问题。

```
用户输入: "预算？"
重写后:   "项目的总预算金额是多少？预算中各项开支的占比如何？"
```

#### 策略二：HyDE（假设文档嵌入）

这是最强大的优化策略。思路很巧妙：

1. 让 AI 先**假装**回答你的问题（生成一段假设性的回答）
2. 用这段假设回答去搜索（而不是用原始问题）

为什么有效？因为"回答"在表述方式上比"问题"更接近文档原文，所以向量搜索更容易命中。

```
用户问题: "Q3 里程碑有哪些？"

HyDE 生成的假设回答:
"第三季度的关键里程碑包括：7 月完成后端 API 开发，
 8 月进行集成测试，9 月 15 日发布 v2.0 正式版……"

用这段假设回答去搜索 → 更容易找到文档中真正描述里程碑的段落
```

#### 策略三：查询分解（Decompose）

把复杂问题拆成多个简单子问题，分别搜索后合并结果。

```
原始问题: "A 方案和 B 方案在成本和时间上有什么区别？"
分解为:
  1. "A 方案的成本是多少？"
  2. "B 方案的成本是多少？"
  3. "A 方案需要多长时间？"
  4. "B 方案需要多长时间？"

四个子问题分别检索，结果合并去重，再由 AI 综合回答
```

### 如何开启

```yaml
retrieval:
  query_processing:
    enabled: true
    strategy: hyde          # 可选: none | rewrite | hyde | decompose
```

### 选哪个好？

| 策略 | 优点 | 缺点 | 推荐场景 |
|------|------|------|---------|
| rewrite | 快速，额外 1 次 LLM 调用 | 提升有限 | 用户经常输入极短的问题 |
| hyde | 检索提升最大 | 额外 1 次 LLM 调用，有时假设偏离实际 | 通用推荐 |
| decompose | 处理复杂多方面问题 | 额外 1+ 次 LLM 调用，速度最慢 | 用户经常问对比类、多维度问题 |

## 十、进阶：自纠错检索（Corrective RAG）

### 问题

有时候检索到的内容和问题不太相关，但系统还是硬着头皮拿去生成回答，导致 AI 要么说"找不到信息"，要么给出牛头不对马嘴的回答。

### 解决方案：CRAG（Corrective RAG）

CRAG 在检索和生成之间加入一个**自我评估环节**。流程如下：

```
提问 → 检索 → AI 评估：这些结果和问题相关吗？
                              │
              ┌───────────────┼────────────────┐
              ▼               ▼                ▼
           相关            不确定            不相关
           直接生成        尝试改进           重写问题
                          生成回答           重新检索
                                               │
                                          （最多循环 N 次）
```

系统会：
1. 检索到结果后，让 AI 判断结果是"相关 / 不确定 / 不相关"
2. 如果不相关，**自动重写查询**（换个问法再搜一次）
3. 最多重试 N 轮（默认 2 轮），最终使用最好的结果来生成回答

### 自适应检索

对话模式下，如果用户的追问不需要搜索文档（比如"谢谢"或"你刚才说的第二点能展开讲讲吗？"），系统会**跳过检索**，直接用对话上下文回答，节省时间。

### 如何开启

```yaml
agentic:
  enabled: true                 # 开启 CRAG
  max_correction_rounds: 2      # 最多纠正几轮
  adaptive_retrieval: true      # 对话模式下自适应跳过检索
```

### 注意事项

- 每轮纠正额外消耗 2 次 LLM 调用（1 次评估 + 1 次重写），**会增加响应时间和 API 费用**
- 对于简单明确的问题（如"XX 是什么？"），基础 RAG 已经足够，不需要 CRAG
- CRAG 最适合文档量大、用户问法多样、检索经常不准的场景

## 十二、常见问题

**Q：开了这么多功能，会不会很慢？**
基础 RAG 通常 2-3 秒响应。混合检索几乎不增加延迟。HyDE 增加 1-2 秒（额外一次 LLM 调用）。CRAG 在最坏情况下增加 5-10 秒（2 轮纠正 × 2 次调用）。建议先开混合检索 + HyDE，观察效果再决定是否开 CRAG。

**Q：使用 Ollama 本地模型时，进阶功能还能用吗？**
完全可以。所有进阶功能（HyDE、CRAG 等）只依赖 LLM 接口，不在乎后端是 OpenAI 还是 Ollama。但本地模型速度较慢，CRAG 的多轮循环延迟会更明显。

**Q：数据安全吗？**
取决于配置。使用 Ollama + 本地 embedding，所有数据都在你的机器上，不会外传。使用 OpenAI，问题和相关文本块会发送到 OpenAI API。

**Q：文档更新了怎么办？**
需要重新上传并索引。系统不会自动检测文件变更。如果使用混合检索，BM25 索引也会同步重建。

**Q：支持中文文档吗？**
支持。分块模块已包含中文识别，BM25 分词器也处理了 CJK 字符。默认的 embedding 模型对中文有基本支持；如果中文文档量大，建议换用多语言模型（如 `paraphrase-multilingual-MiniLM-L12-v2`）。

**Q：最多能处理多少文档？**
FAISS 向量数据库在内存中运行，百万级文本块仍可毫秒级搜索。实际瓶颈通常在 embedding 生成速度（本地模型约 100 块/秒）和内存大小。

## 十三、故障排除

### 上传文档后 Indexed Chunks 仍然是 0

- 检查文件是否为空或损坏
- PDF 如果是扫描件（图片），需要先 OCR 转文字
- 查看终端日志中是否有报错

### 回答总是说"找不到相关信息"

- 确认文档已成功索引（左侧 Indexed Chunks > 0）
- 尝试调高 Top-K（比如从 5 改到 10）
- 尝试开启混合检索，增加关键词匹配能力
- 问题太模糊？开启查询优化（HyDE 或 rewrite）

### API 返回 429 Too Many Requests

系统有内置速率限制（默认 30 请求/分钟）。调整环境变量：

```bash
RATE_LIMIT_MAX=100 streamlit run app.py
```

### 内存不足

- 减小 `chunk_size`（默认 512，可改为 256）
- 使用 ChromaDB 替代 FAISS（ChromaDB 会持久化到磁盘）
- 减小 embedding 模型的 `batch_size`
