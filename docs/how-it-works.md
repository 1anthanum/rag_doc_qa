# RAG Document Q&A — 系统工作原理

> 这份文档用通俗的语言解释本项目的核心流程，帮助没有 AI/NLP 背景的读者快速理解系统是如何工作的。

---

## 一句话概括

你上传一些文档，然后用自然语言提问，系统会从文档中找到相关内容，再用 AI 生成一个有据可查的回答。

---

## 为什么需要 RAG？

你可能用过 ChatGPT 之类的 AI，它们很聪明，但有一个致命缺点：它们只知道训练时见过的内容，对你的私有文档一无所知。

RAG（Retrieval-Augmented Generation，检索增强生成）解决的就是这个问题。它的思路很简单：先从你的文档里**找到**相关段落，再把这些段落**喂给** AI，让 AI 基于这些内容来回答。这样 AI 的回答就有了依据，不会凭空编造。

打个比方：普通 AI 像一个博学但没读过你的报告的专家；RAG 则是先让这个专家翻阅你的报告，再回答你的问题。

---

## 完整流程：从上传到回答

整个系统分为三个阶段，像一条流水线：

```
文档上传 ──→ 【阶段一：摄入】 ──→ 【阶段二：检索】 ──→ 【阶段三：生成】 ──→ 回答
```

下面逐一拆解。

---

### 阶段一：文档摄入（Ingestion）

**目标：** 把你的文档变成计算机能快速搜索的格式。

这个阶段有三步：

#### 第 1 步：加载文档

系统接收你上传的文件（PDF、Word、TXT、Markdown），用对应的解析器提取出纯文本内容。比如 PDF 会用 PyPDF2 库逐页提取，Word 会用 python-docx 提取段落。

这一步的产出是一个 `Document` 对象，包含文本内容和来源信息（文件名、大小等）。

#### 第 2 步：文本分块（Chunking）

一篇文档可能有几千甚至几万字，直接丢给 AI 处理既慢又贵，而且 AI 对长文本的理解能力也有限。所以系统会把长文本切成小块（默认每块约 512 个字符），相邻的块之间有少量重叠（64 个字符），确保不会在句子中间硬切断。

系统支持三种切分策略：

| 策略 | 工作方式 | 适用场景 |
|------|---------|---------|
| 固定长度 | 按字符数机械切分 | 简单快速，但可能切断句子 |
| 句子边界 | 在句号、问号等位置切分 | 保持语义完整性 |
| 递归切分（默认） | 先尝试按段落切，段落太长再按句子切 | 最智能，效果最好 |

每个切出的块（`Chunk`）都会记住自己来自哪个文档的哪个位置，方便后面追溯来源。

#### 第 3 步：向量化（Embedding）

这是整个流程中最关键也最"魔法"的一步。

计算机不能直接理解文字的含义，但可以理解数字。**向量化**就是把一段文字转换成一串数字（向量），使得意思相近的文字对应的数字也相近。

举个例子：

- "猫在沙发上睡觉" → `[0.23, -0.41, 0.87, ...]`（384 个数字）
- "小猫躺在椅子上" → `[0.25, -0.39, 0.85, ...]`（很接近！）
- "今天股市大涨" → `[-0.72, 0.15, -0.33, ...]`（差别很大）

系统支持两种向量化方式：

- **本地模型**（默认）：使用 `all-MiniLM-L6-v2`，免费，在你的电脑上运行，不需要网络。生成 384 维向量。
- **OpenAI API**：使用 `text-embedding-3-small`，需要 API Key 和网络，效果更好。生成 1536 维向量。

向量化后的文本块被存入**向量数据库**（默认使用 FAISS），方便后续快速搜索。

> **小结：** 经过这三步，你的文档从"一堆文字"变成了"向量数据库中可以快速检索的数字块"。

---

### 阶段二：检索（Retrieval）

**目标：** 当用户提问时，找到文档中最相关的几个片段。

当你输入一个问题，比如"项目的预算是多少？"，系统会：

1. **问题向量化：** 用同样的模型把你的问题也转换成一串数字。
2. **相似度搜索：** 在向量数据库中找到和你的问题"数字上最接近"的文本块。这就像在一个巨大的数字空间里找距离最近的邻居。默认返回最相关的 5 个块。
3. **（可选）重排序：** 如果开启了重排序功能，系统会用一个更精确的模型（CrossEncoder）对初步结果再排一次序，提高准确率。

> **类比：** 这个过程就像你在图书馆找资料——你不会把整个图书馆的书都读一遍，而是先通过索引找到最可能包含答案的几页，然后只看那几页。

---

### 阶段三：生成（Generation）

**目标：** 基于检索到的内容，用 AI 生成一个清晰、有依据的回答。

系统将检索到的文本块和你的问题组装成一个精心设计的提示词（Prompt），发送给大语言模型（LLM）。提示词的核心指令是：

- **只能**基于提供的上下文回答
- 如果上下文中找不到答案，必须明确说明
- 必须标注信息来自哪个来源

系统支持四种问答模式：

| 模式 | 用途 | 示例 |
|------|------|------|
| QA（问答） | 针对性回答具体问题 | "项目截止日期是什么时候？" |
| 摘要 | 总结检索到的内容要点 | "帮我总结这份报告的核心发现" |
| 对比 | 比较不同来源的信息 | "这两份报告对市场趋势的看法有何不同？" |
| 对话 | 多轮对话，记住上下文 | 连续追问，系统记住之前的问答 |

AI 回答后，系统会把回答连同来源列表一起返回给你。你可以看到每条信息出自哪个文档的哪个片段，方便验证。

## 系统架构总览

```
用户
 │
 ├──→ Streamlit 网页界面（上传文档 + 聊天窗口）
 │
 └──→ FastAPI REST API（程序化接口）
       │
       ▼
   RAGChain 编排器
       │
       ├──→ 摄入管线：加载 → 分块 → 向量化
       ├──→ 检索管线：问题向量化 → 相似搜索 → 重排序
       └──→ 生成管线：组装提示词 → 调用 LLM → 返回回答
```

系统提供两种使用方式：

- **Streamlit 网页界面：** 适合非技术用户，打开浏览器就能用。左边上传文档，右边聊天提问。
- **FastAPI REST API：** 适合开发者，可以通过代码调用，集成到其他系统中。

## 关键技术选型

| 组件 | 默认选项 | 替代选项 | 选择理由 |
|------|---------|---------|---------|
| 向量化模型 | all-MiniLM-L6-v2（本地） | OpenAI API | 本地免费且速度快 |
| 向量数据库 | FAISS | ChromaDB | FAISS 轻量高效，适合中小规模 |
| 大语言模型 | OpenAI GPT-4o-mini | Ollama（本地） | GPT 效果好；Ollama 可完全离线 |
| 切分策略 | 递归切分 | 固定长度 / 句子边界 | 递归切分在多数场景下效果最优 |

## 一个完整的例子

假设你上传了一份 20 页的项目计划书（PDF），然后问："第三季度的里程碑有哪些？"

1. **加载：** 系统从 PDF 提取出全部文本（约 8000 字）
2. **分块：** 文本被切成约 30 个小块，每块约 500 字符
3. **向量化：** 30 个块各自变成一串 384 维的数字，存入 FAISS
4. **问题向量化：** "第三季度的里程碑有哪些？" → 一串 384 维的数字
5. **搜索：** FAISS 找到最相关的 5 个块（可能来自"项目时间线"和"Q3 交付计划"部分）
6. **生成：** 这 5 个块 + 你的问题一起发给 GPT-4o-mini
7. **回答：** AI 返回："根据项目计划书，第三季度的里程碑包括：……（来源：第 12 页、第 15 页）"

整个过程通常只需要几秒钟。

## 常见问题

**Q：系统会把我的文档发送到外部服务器吗？**
取决于你的配置。如果使用本地模型（Ollama + sentence-transformers），所有数据都留在你的电脑上。如果使用 OpenAI，问题和相关文本块会发送到 OpenAI 的 API。

**Q：支持中文文档吗？**
支持。句子切分模块已包含中文句号识别，向量化模型对中文也有基本支持。如果中文文档较多，建议使用支持多语言的向量化模型。

**Q：文档更新了怎么办？**
目前需要重新上传并索引。系统不会自动检测文件变更。

**Q：一次能处理多大的文档？**
默认单文件上限 50 MB。向量数据库（FAISS）在内存中运行，百万级别的文本块仍可高效搜索。

**Q：回答不准确怎么办？**
可以尝试：调高 Top-K 值（检索更多片段）、开启重排序功能、使用更好的向量化模型、或检查文档本身是否包含所需信息。

## 进阶功能

本文档介绍了基础 RAG 流程。系统还支持以下 SOTA 进阶功能：

- **混合检索（Hybrid Search）** — BM25 关键词匹配 + 向量语义搜索，双管齐下
- **语义分块（Semantic Chunking）** — 在话题转换处切分，产出更连贯的文本块
- **查询优化（HyDE / 重写 / 分解）** — 让 AI 先优化你的问题再去搜索
- **自纠错检索（Corrective RAG）** — AI 评估检索结果，不满意自动换方式重搜

详细说明请阅读 **[新手完全教程](./tutorial.md)**。
